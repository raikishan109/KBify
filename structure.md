# KBify Project Structure & Architecture

## 🏙️ High-Level Architecture
KBify has been refactored from a monolithic application into a **Reactive, Data-Driven SaaS Architecture**. This modular structure separates data (Registry), logic (Tools), UI updates (Navigation), and state (Store).

## 🛠️ Core Systems

### 1. The Registry (`registry.js`)
The `registry.js` file is the **Single Source of Truth** for the entire application. It contains all tool configurations, icons, and section metadata. 
*   **Scalability**: Adding a new tool only requires 1 entry in this file. The UI generates itself automatically.

### 2. Reactive Store (`state.js`)
Uses an **ES6 Proxy** to watch for state changes. When a property changes (e.g., `theme`, `activeTool`, `progress`), the store automatically:
*   Persists the data to `localStorage`.
*   Notifies all registered UI listeners through a `subscribe` system.

### 3. Dynamic Rendering (`navigation.js`)
Instead of static HTML, the tool cards are generated at runtime by `initToolGrid()`. This ensures that tool availability and design are always synchronized with the code.

### 4. UI/UX Components
*   **Toast System**: Modular notifications for errors/success.
*   **Refined Modal**: Professional display for final compression results.
*   **Responsive Sidebar**: Smooth, gesture-friendly navigation.

## 🚀 Production Optimizations
*   **Event Delegation**: Minimizes memory usage by using document-level listeners.
*   **DOM Caching**: Uses a unified `UI` cache object to prevent redundant `getElementById` calls.
*   **PWA Ready**: Supports offline mode, installation, and instant updates.
*   **Binary Search Compression**: (In Image Tool) Fast, precise quality matching for JPG/PNG.
